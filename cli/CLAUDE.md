# Analysi CLI

Config-driven CLI for the Analysi Security Platform, built on oclif + @clack/prompts.

## Quick Start

```bash
make cli-install       # install deps (first time only)
make cli-build         # compile TypeScript
make cli CMD="--help"  # run
```

## Architecture

```
cli-config.yaml → generate-commands.ts → thin oclif command files → base-command.ts → API
```

The pipeline is: edit YAML → run codegen → build. Everything flows through `base-command.ts` which handles auth, API calls, error handling, and output formatting.

### Two kinds of commands

1. **Generated commands** — defined in `cli-config.yaml`, generated by `scripts/generate-commands.ts`. These are thin ~30-line files that just declare args/flags and call `this.apiCall()`.
2. **Hand-written commands** — `auth/login.ts`, `config/*`, `status.ts`, `alerts/analyze.ts`, `alerts/watch.ts`, `alerts/validate.ts`, `tasks/run.ts`, `tasks/create.ts`, `tasks/update.ts`, `tasks/compile.ts`, `tasks/run-adhoc.ts`, `workflows/run.ts`, `workflows/compose.ts`, `workflow-runs/watch.ts`, `integrations/run-tool.ts`, `tools/get.ts`. These use `@clack/prompts` for interactive flows, make multiple parallel API calls, or implement live progress tracking. The generator skips directories in `PROTECTED_DIRS` and individual files in `PROTECTED_FILES`.

### Key design decisions

- **Config-driven codegen over hand-written commands.** Adding a new API endpoint to the CLI is a YAML edit + `make cli-generate`. No TypeScript needed. The generator handles imports, flag types, path resolution, body parsing — everything.
- **Smart output defaults.** Table output auto-selects up to 8 columns using `priorityKeys` (IDs, names, status first) and `skipKeys` (large JSON blobs). Users override with `--fields`. This means `alerts list` is useful out of the box without knowing the schema.
- **Relative time in tables, raw timestamps in JSON/CSV.** Tables show "3m ago" for quick scanning. JSON and CSV preserve exact ISO timestamps for scripting. This is the right tradeoff: humans read tables, machines read JSON.
- **Status and severity are colorized.** `completed` = green, `failed` = red, `running` = yellow. `high` = red, `medium` = yellow. Makes visual scanning instant.
- **`@filepath` syntax for data input.** `--data @input.json` reads from file, `--data '{"key":"val"}'` parses inline JSON. Handled by `data-reader.ts`.
- **Connection errors are caught.** Network failures (ECONNREFUSED) show actionable guidance instead of stack traces.
- **`--out <file>`** writes output to a file (confirmation goes to stderr, keeping stdout clean for piping).

## Adding a New Command

1. Edit `cli-config.yaml` — add the operation under the appropriate topic:
   ```yaml
   commands:
     alerts:
       operations:
         my-new-command:
           description: Do something new
           method: GET
           path: /alerts/{alert_id}/something
           args:
             alert_id: { required: true, description: Alert ID }
           flags:
             limit: { type: integer, default: 25, description: Max results }
             status:
               type: string
               description: Filter by status
               options: [pending, completed, failed]  # enables enum validation
   ```
2. Run `make cli-generate`
3. Run `make cli-build`

For flags with a fixed set of valid values, add `options: [...]` — oclif validates automatically.

## Adding a New Topic

1. Add the topic to `cli-config.yaml` under `commands:`
2. Add the topic to `package.json` → `oclif.topics` for the help description
3. Run `make cli-generate && make cli-build`

## Adding a Hand-Written Command

Create the file directly in `src/commands/`. It won't be overwritten by the generator. Extend `BaseCommand` for API commands, or `Command` for standalone commands. See `status.ts` as an example of a hand-written command that makes parallel API calls.

If the command lives under a new protected directory, add it to `PROTECTED_DIRS` in `generate-commands.ts`. For individual files in otherwise-generated directories (like `alerts/analyze.ts`), add to `PROTECTED_FILES` instead.

## Output System (`output.ts`)

The output system has three layers:

1. **`printResponse()`** — entry point. Routes to table/JSON/CSV based on `PrintOptions`.
2. **`printTable()`** — array data. Smart column selection, ANSI-aware truncation, status/severity coloring, relative timestamps.
3. **`printObject()`** — single object. Key-value pairs with truncated previews for large nested values.

### Column selection logic

- `priorityKeys` — always shown first: `id`, `alert_id`, `name`, `status`, `severity`, `created_at`, etc.
- `skipKeys` — never shown in table: `raw_alert`, `tenant_id`, `cy_script`, etc. (large JSON blobs or redundant)
- Max 8 columns per table for terminal readability
- `--fields col1,col2` overrides all of this

### Color maps

Status: `completed`/`healthy` = green, `running`/`analyzing`/`pending` = yellow, `failed`/`unhealthy` = red
Severity: `critical` = bold red, `high` = red, `medium` = yellow, `low` = blue

## Base Flags (available on all generated commands)

| Flag | Short | Description |
|------|-------|-------------|
| `--tenant` | `-t` | Tenant ID (or `ANALYSI_TENANT_ID` env var) |
| `--output` | `-o` | `table` (default), `json`, `csv` |
| `--fields` | | Comma-separated column list |
| `--no-header` | | Suppress table/CSV header (scripting) |
| `--out` | | Write output to file |
| `--verbose` | `-v` | Show request URL, query params, timing |

## Auth

Credentials stored at `~/.config/analysi/credentials.json` with `0600` permissions. Tenant resolution order:
1. `--tenant` flag
2. `ANALYSI_TENANT_ID` env var
3. Default from `analysi auth login`

## Testing

```bash
make cli-test          # run unit tests (vitest)
make cli-generate      # regenerate from YAML
make cli-build         # compile TypeScript (fails on type errors)
make cli CMD="status"  # quick smoke test
make cli CMD="alerts list --verbose"  # verify with debug output
```

## Common Patterns

```bash
# Scripting: extract alert IDs as plain text
analysi alerts list --output csv --fields alert_id --no-header

# Debug an API call
analysi alerts get <id> --verbose

# Save report to file
analysi alerts list --output json --out alerts.json

# Read input from file
analysi tasks run <task_id> --data @input.json
```
