# Suspicious Login from Unusual Location — Investigation Runbook

> ⚠️ **COMPOSED RUNBOOK** — Confidence: LOW
> Composed from `forced-authentication-detected.md` (score: 45) with Identity-specific adaptations.
> This runbook covers all alerts generated by the `unusual_login_location` detection rule across Okta and other Identity providers.
> **Analyst review required before operational use.**

---

## Overview

This runbook investigates alerts triggered when a user authenticates from a geographically unusual or anomalous location. This pattern is associated with:
- **T1078** — Valid Accounts (attacker using stolen/compromised credentials)
- **T1133** — External Remote Services (legitimate accounts accessed from unexpected external IPs)
- **T1556** — Modify Authentication Process (potential MFA bypass or policy circumvention)

**Key Question:** Was the login performed by the legitimate account owner, or by an attacker using stolen credentials?

---

## Steps

### 1. Alert Understanding & Initial Assessment ★

- **Action:** Analyze alert and form investigation hypotheses
- **Pattern:** hypothesis_formation
- **Input:** finding_info.title, severity, get_primary_user(alert) (email), get_src_ip(alert), unmapped.country, unmapped.normal_country, finding_info.desc
- **Outputs:**
  - `context_summary`: Human-readable alert summary including user, source country, and normal country
  - `investigation_hypotheses`: List of theories (e.g., account takeover, travel, VPN usage, shared credentials)
  - `key_observables`: Key indicators — user identity, source IP, source country, authentication timestamp

**Hypotheses to form:**
1. **Credential compromise / ATO** — Attacker obtained valid credentials and logged in from foreign IP
2. **Legitimate travel** — User genuinely traveling in flagged country
3. **VPN / Proxy usage** — User behind anonymizing infrastructure resolving to unexpected country
4. **Shared or delegated access** — Third-party accessing account on user's behalf
5. **Impossible travel** — Authentication physically impossible given prior login location/time

---

### 2. User Account Baseline & History Analysis ★

- **Action:** Retrieve the user's historical login patterns from the Identity provider
- **Purpose:** Establishes normal behavior baseline to validate the anomaly
- **Pattern:** integration_query
- **Integration:** identity (Okta / SIEM)
- **Focus:** Last 30 days of successful logins — countries, IP addresses, device fingerprints, login times
- **Fields:** get_primary_user(alert), time
- **Output:** `user_login_history`
- **Decision Points:**
  - `has_prior_logins_from_same_country` → Potentially benign (VPN or regular travel)
  - `first_ever_login_from_this_country` → High suspicion, proceed with escalation track
  - `multiple_failed_logins_before_success` → Credential stuffing or brute force attempt
  - `no_login_history_at_all` → Newly created or dormant account — high risk

---

### 3. Concurrent Evidence Collection ★

**Run all sub-steps in parallel:**

#### 3a. Impossible Travel Check ★

- **Action:** Check for physically impossible authentication between the alert login and the user's most recent prior login
- **Purpose:** Validates "Legitimate travel" vs "Credential compromise"
- **Pattern:** payload_analysis
- **Depends On:** Step 2
- **Input:** `outputs.user_login_history` (last login timestamp + location)
- **Focus:** Calculate time delta and physical distance between previous login location and current login location
- **Output:** `impossible_travel_result`
- **Decision Points:**
  - `travel_physically_impossible (speed > 1000 km/h)` → Strong indicator of ATO — escalate immediately
  - `travel_plausible_but_unusual` → Investigate further with user verification
  - `no_prior_login_data` → Cannot assess — treat as high risk

#### 3b. Source IP Characterization ★

- **Action:** Identify the nature of the source IP address
- **Purpose:** Distinguish attacker infrastructure from user VPN/travel
- **Pattern:** integration_query
- **Integration:** threat_intel / siem
- **Fields:** get_src_ip(alert)
- **Output:** `ip_characterization`
- **Decision Points:**
  - `known_tor_exit_node` → High risk — attacker anonymizing traffic
  - `known_vpn_provider` → May be legitimate VPN use — verify with user
  - `residential_isp_in_flagged_country` → Higher risk of genuine foreign access
  - `datacenter_or_hosting_provider` → Likely automated or infrastructure-based attack
  - `ip_on_threat_intel_blocklist` → Confirmed malicious — escalate

#### 3c. Okta / Identity Provider Authentication Details ★

- **Action:** Retrieve full authentication event details from Okta (or relevant IdP)
- **Purpose:** Capture device, MFA status, session details, and risk signals
- **Pattern:** integration_query
- **Integration:** identity (Okta)
- **Fields:** get_primary_user(alert), time, get_src_ip(alert)
- **Output:** `auth_event_details`
- **Decision Points:**
  - `mfa_challenged AND passed` → User has MFA; attacker may have bypassed or phished MFA token
  - `mfa_not_challenged` → MFA absent or bypassed — critical gap
  - `new_device_fingerprint` → First-time device for this user — high suspicion
  - `okta_risk_score >= HIGH` → Platform's own risk engine flagged the event
  - `session_created == true` → Active session established — immediate response may be needed

### 4. Threat Intelligence Enrichment

- **Parallel:** Yes
- **Optional:** Both sources can fail gracefully

#### IP Reputation Check

- **Action:** Check attacker IP reputation in threat intelligence sources
- **Purpose:** Validates threat actor profile and attribution
- **Pattern:** integration_query
- **Integration:** threat_intel
- **Condition:** IF threat_intel configured
- **Fields:** get_src_ip(alert)
- **Output:** `ip_reputation`

#### IP Abuse History & Geolocation

- **Action:** Check IP abuse history and geographic location from threat databases
- **Purpose:** Context for attacker origin and history
- **Pattern:** integration_query
- **Integration:** threat_intel
- **Condition:** IF threat_intel configured
- **Fields:** get_src_ip(alert)
- **Output:** `geo_abuse_data`

---

### 5. MFA Bypass & Session Hijack Assessment

- **Action:** Assess whether MFA was circumvented and whether the session poses ongoing risk
- **Depends On:** Steps 3b, 3c
- **Pattern:** payload_analysis
- **Input:** `outputs.auth_event_details`, `outputs.ip_characterization`
- **Focus:** MFA challenge outcome, session token age, concurrent sessions from different locations
- **Output:** `mfa_session_assessment`
- **Decision Points:**
  - `mfa_bypassed OR not_enforced` → Policy violation — escalate and consider immediate MFA enforcement
  - `concurrent_sessions_from_different_geo` → Active session hijack in progress
  - `session_age_very_short AND high_activity` → Automated or scripted access
  - `oauth_token_issued` → Check for downstream app access (email, files, cloud resources)

---

### 6. Authorized Travel / VPN Verification

- **Action:** Contact user or check travel/VPN authorization records to rule out benign causes
- **Purpose:** Validates "Malicious access" vs "Authorized unusual access"
- **Pattern:** integration_query
- **Integration:** email
- **Condition:** IF email integration configured OR if escalation threshold not yet reached
- **Fields:** get_primary_user(alert), time, unmapped.country
- **Output:** `user_verification_result`
- **Decision Points:**
  - `user_confirms_travel` → Benign — document and close with recommendation for trusted location setup
  - `user_confirms_vpn_usage` → Benign — recommend whitelisting or trusted network policy update
  - `user_denies_login` → Confirmed ATO — escalate immediately, initiate IR process
  - `user_unreachable` → Treat as suspected ATO pending confirmation

---

### 7. Post-Authentication Activity Review ★

- **Action:** Enumerate all actions taken within the suspicious session
- **Depends On:** Step 3c
- **Condition:** IF `outputs.auth_event_details.session_created == true`
- **Pattern:** integration_query
- **Integration:** siem / identity / edr
- **Focus:** File access, email access, password changes, MFA device registration, OAuth app authorizations, privilege changes, data exports
- **Fields:** get_primary_user(alert), session_id (from auth_event_details)
- **Output:** `session_activity`
- **Decision Points:**
  - `mfa_device_registered_in_session` → Attacker establishing persistence — critical
  - `password_changed_in_session` → Account takeover confirmed — lockout immediately
  - `sensitive_data_accessed` → Data exfiltration risk — escalate to DLP review
  - `admin_role_added` → Privilege escalation — critical escalation
  - `oauth_app_authorized` → Check app permissions; attacker may maintain access post-password-reset

---

### 8. Account Compromise Determination ★

- **Action:** Synthesize all evidence to determine if this represents a true account takeover
- **Depends On:** Steps 2, 3a, 3b, 3c, 4, 5, 6, 7
- **Pattern:** impact_assessment
- **Input:** `outputs.user_login_history`, `outputs.impossible_travel_result`, `outputs.ip_characterization`, `outputs.auth_event_details`, `outputs.mfa_session_assessment`, `outputs.user_verification_result`, `outputs.session_activity`
- **Output:** `compromise_verdict`
- **Decision Points:**
  - `impossible_travel AND user_denies_login` → **CONFIRMED ATO** — immediate escalation
  - `tor_or_known_malicious_ip AND new_device AND no_mfa` → **HIGH CONFIDENCE ATO** — escalate
  - `session_activity includes persistence_indicators` → **ACTIVE INTRUSION** — emergency response
  - `user_confirms_travel AND vpn_explains_geo` → **BENIGN** — close with documentation
  - `inconclusive_evidence AND high_risk_signals` → **SUSPECTED ATO** — escalate for manual review

---

### 9. Hypothesis Validation ★

- **Action:** Match evidence against initial hypotheses from Step 1
- **Depends On:** All prior steps
- **Pattern:** evidence_correlation
- **Input:** `outputs.investigation_hypotheses`, `outputs.compromise_verdict`, `outputs.impossible_travel_result`, `outputs.ip_reputation`, `outputs.user_verification_result`, `outputs.session_activity`
- **Output:** `validated_hypothesis`

## Conditional Logic

### Branch: Confirmed Account Takeover (ATO)

- **Condition:** `outputs.compromise_verdict == "CONFIRMED_ATO" OR outputs.user_verification_result.user_denies_login == true`
- **Immediate Actions (in order):**
  1. Terminate all active sessions for the user in the Identity provider
  2. Force password reset (admin-initiated)
  3. Revoke all OAuth tokens and app authorizations
  4. Remove any MFA devices registered during the suspicious session
  5. Escalate to Tier 2 / Incident Response team
  6. Notify the affected user via out-of-band channel (phone, secondary email)
  7. Review all session activity for data exfiltration
  8. Check for lateral movement to other accounts (same credentials reused)
  9. Assess if privileged account — if yes, treat as critical incident

### Branch: Suspected ATO (Inconclusive)

- **Condition:** `outputs.compromise_verdict == "SUSPECTED_ATO"`
- **Steps:**
  - Escalate to Tier 2 with full evidence package
  - Place account under enhanced monitoring (step-up authentication)
  - Attempt user verification via phone / manager
  - Do NOT terminate session yet — monitor for 30 minutes while investigating
  - If user unreachable after 2 hours → treat as Confirmed ATO

### Branch: Benign (Travel / VPN)

- **Condition:** `outputs.user_verification_result.confirmed_benign == true`
- **Steps:**
  - Document user explanation and supporting evidence
  - Recommend user configure trusted locations in Okta
  - Consider adding VPN IP range to allowlist if recurring
  - Close alert as Benign with full documentation
  - Recommend MFA enforcement review if MFA was not challenged

### Branch: MFA Persistence Established

- **Condition:** `outputs.session_activity.mfa_device_registered == true`
- **Additional Steps (priority — run immediately):**
  - Remove attacker-registered MFA device before password reset
  - Audit all MFA device registrations for the past 30 days across the organization
  - Check for other accounts where this IP registered MFA devices
  - Escalate to security engineering for policy review

### Branch: OAuth / App Authorization Issued

- **Condition:** `outputs.session_activity.oauth_app_authorized == true`
- **Additional Steps:**
  - Enumerate all OAuth applications authorized in the suspicious session
  - Review scopes requested by each application
  - Revoke OAuth tokens for all apps authorized during the session
  - Check if apps have offline_access scope (survive password reset)
  - For high-permission apps — treat as critical and escalate

### Branch: Privileged Account

- **Condition:** `get_primary_user(alert)` is a privileged/admin account (verify via directory)
- **Additional Steps:**
  - Treat with maximum severity regardless of other factors
  - Immediate containment — do not wait for user confirmation
  - Escalate to CISO/Security Leadership
  - Review admin actions performed in the session
  - Check for new admin account creation or permission grants

## Investigation Notes

### Key Evidence Sources

1. **Okta System Logs / Identity Provider Logs:** Primary source — authentication events, device fingerprints, MFA outcomes, session details, user activity
2. **SIEM (aggregated logs):** Corroborate authentication events, correlate with other anomalies in the environment
3. **Threat Intelligence:** IP reputation, Tor/VPN/proxy classification, geolocation accuracy
4. **Email/ITSM (if available):** User travel confirmation, helpdesk notes, IT asset records
5. **EDR (if available):** Post-authentication endpoint activity if session was used to access managed devices
6. **Directory Service (AD/LDAP):** Account privilege level, group membership, recent changes

### Common Attack Patterns

- **Credential Stuffing:** Attacker uses credentials from previous breaches; single successful login from foreign IP with no prior failures in IdP (failures happened at source, not target)
- **Phishing / AiTM (Adversary-in-the-Middle):** Attacker captures session cookie bypassing MFA; look for new device + MFA passed without challenge or with very fast approval
- **Password Spray:** Multiple accounts from same IP around same time — check for lateral alerting
- **Sold/Leaked Credentials:** Credentials purchased from dark web; often has no preceding failed attempts
- **Insider Threat:** Employee sharing credentials or accessing from unauthorized location; user may confirm access was theirs but unauthorized by policy

### Investigation Efficiency

- Steps 3a, 3b, 3c can run fully in parallel (different data sources)
- Step 4 (threat intel) can run in parallel with Step 3
- Step 6 (user verification) can begin in parallel with steps 5 and 7
- **Critical path:** 1 → 2 → 3a,3b,3c → 8 → 9 → Final Analysis
- **Fast escalation trigger:** If `impossible_travel_result.impossible == true` → skip straight to Step 8 and escalate

### Critical Decision: When to Escalate Immediately

- **ALWAYS escalate immediately if:**
  - Impossible travel confirmed
  - User denies the login
  - MFA device registered in suspicious session
  - Privileged/admin account
  - Active session with ongoing high-risk activity
- **Monitor and investigate if:**
  - High-risk IP but user hasn't been reached yet
  - Unusual country but possible business travel
- **Lower priority (but document):**
  - User confirms VPN usage
  - Known residential ISP with no threat intel hits and user confirms travel

### Security Recommendations

- Enforce MFA for all users without exception
- Implement Okta ThreatInsight or equivalent risk-based authentication
- Enable impossible travel detection policies in the Identity provider
- Restrict access from Tor exit nodes and known hosting/proxy ranges
- Enforce conditional access policies (require trusted device or network)
- Implement session duration limits for high-risk authentication events
- Train users to report unfamiliar MFA push requests immediately
- Consider hardware token (FIDO2/WebAuthn) for privileged accounts
- Review and enforce OAuth app consent policies to limit scope creep
