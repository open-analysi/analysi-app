# SIEM Alert Investigation Reference

## Overview

SIEM (Security Information and Event Management) alerts are notifications generated based on predefined rules and correlation algorithms. These rules are customized to meet an organization's specific security requirements.

## How SIEM Alerts Work

### Detection

When an event matches defined criteria in the SIEM system, an alert is triggered. These alerts are:
- Prepared by cybersecurity experts
- Continuously updated
- Cover range of threats from common attacks to sophisticated zero-day exploits
- Based on correlation of multiple log sources

### Alert Generation Process

1. **Log Collection**: SIEM aggregates logs from multiple sources (firewalls, IDS/IPS, EDR, proxies, etc.)
2. **Normalization**: Data is standardized into common format
3. **Correlation**: Events are correlated based on rules and algorithms
4. **Alert Triggering**: When correlation matches rule criteria, alert is generated
5. **Analyst Assignment**: Alert is assigned to analyst for investigation

## Key Alert Fields

### Standard Alert Information

**Severity**
- Categorized as: Low, Medium, High, or Critical
- Indicates urgency and potential impact
- Helps prioritize investigation efforts

**Date**
- Timestamp of the incident
- Usually in UTC +0 timezone
- Critical for timeline reconstruction

**Rule Name**
- Identifies specific rule triggered by event
- Example: "SOC138 - Detected Suspicious Xls File"
- Provides context for what detection logic fired

**EventID**
- Unique identifier for the event
- Used for tracking and case management
- Allows correlation across systems

**Type**
- Specifies type of event or alert
- Examples: Malware, Phishing, Web Attack, Network Anomaly
- Determines investigation approach

**Action**
- Assignment status (who owns the alert)
- Next steps required
- Can indicate automated actions taken (blocked, allowed, quarantined)

## Playbooks

### What are Playbooks?

**Definition**: Workflows for effective and consistent analysis of alerts generated by SIEM or other security tools.

**Purpose**: Provide step-by-step processes to follow during investigation.

### Why Playbooks are Important

1. **Consistency**: Ensures all analysts follow same investigation procedures
2. **Efficiency**: Reduces investigation time by providing clear steps
3. **Knowledge Transfer**: Helps new analysts learn investigation techniques
4. **Completeness**: Prevents missing critical investigation steps
5. **Documentation**: Creates standardized investigation records

### Common Playbook Types

Different alert types require different investigation approaches:

**Web Attacks Playbook**
- Check source IP reputation
- Analyze HTTP request parameters
- Verify if attack successful (response codes)
- Check WAF/IPS logs for blocking
- Investigate affected web server

**Ransomware Playbook**
- Isolate affected endpoint immediately
- Check for file encryption activity
- Identify ransomware family
- Search for lateral movement
- Check backup integrity

**Malware Playbook**
- Verify quarantine/cleanup status
- Analyze file hash in VirusTotal
- Check for C2 communication
- Investigate process execution
- Search for persistence mechanisms

**Phishing Playbook**
- Verify email authenticity (SPF, DKIM, DMARC)
- Analyze attachments and URLs
- Check if users clicked links
- Search mail gateway for similar emails
- Delete emails from other inboxes if malicious

**Network Anomaly Playbook**
- Identify source and destination
- Check for known IOCs
- Analyze traffic patterns
- Investigate user/device behavior
- Correlate with other alerts

## General Alert Investigation Framework

### Phase 1: Initial Assessment (5 minutes)

1. **Read alert details completely**
   - Understand what triggered the alert
   - Identify severity and type
   - Note timestamp and affected assets

2. **Quick context gathering**
   - Check if this is known false positive
   - Review similar historical alerts
   - Identify affected user/system criticality

3. **Determine urgency**
   - Critical severity = immediate investigation
   - Check if attack is ongoing
   - Assess potential business impact

### Phase 2: Data Collection (10-15 minutes)

1. **Gather relevant logs**
   - Firewall logs for network activity
   - Proxy logs for web traffic
   - EDR logs for endpoint activity
   - Authentication logs for user activity
   - Email logs for phishing alerts

2. **Correlate events**
   - Search for related activity from same source
   - Check for activity to same destination
   - Look for similar patterns in timeframe
   - Identify if part of larger campaign

3. **Extract IOCs**
   - IP addresses (source and destination)
   - Domains and URLs
   - File hashes
   - User accounts
   - Email addresses

### Phase 3: Analysis (15-30 minutes)

1. **Validate the alert**
   - Determine if true positive or false positive
   - Verify detection logic is sound
   - Check for business context explaining activity

2. **Assess impact**
   - Was attack successful?
   - What data/systems were accessed?
   - Is there ongoing activity?
   - Potential for lateral movement?

3. **Determine scope**
   - How many systems affected?
   - How many users involved?
   - Timeline of activity
   - Related alerts or incidents

### Phase 4: Response (varies)

1. **Containment actions**
   - Block malicious IPs/domains
   - Isolate compromised endpoints
   - Disable compromised accounts
   - Delete malicious emails

2. **Escalation (if needed)**
   - Criteria for escalation in triage-procedures.md
   - Notify appropriate teams
   - Provide clear summary of findings
   - Include all IOCs and evidence

3. **Documentation**
   - Document all investigation steps
   - Record findings and confidence level
   - List IOCs discovered
   - Note actions taken
   - Provide recommendations

### Phase 5: Closure

1. **Final classification**
   - True Positive (confirmed malicious)
   - False Positive (benign activity)
   - Inconclusive (insufficient data)

2. **Update ticketing system**
   - Close alert with appropriate status
   - Add detailed notes
   - Update knowledge base if needed

3. **Post-investigation tasks**
   - Tune detection rules if false positive
   - Share IOCs with threat intelligence
   - Update playbooks with lessons learned

## Best Practices for SIEM Alert Investigation

### Do's

- **Follow the playbook**: Use established procedures for consistency
- **Document everything**: Maintain detailed timestamped notes
- **Think critically**: Don't blindly trust automated classifications
- **Correlate widely**: Check multiple log sources
- **Preserve evidence**: Take screenshots, save logs
- **Ask for help**: Escalate when uncertain rather than making assumptions

### Don'ts

- **Don't rush**: Thorough investigation prevents missing critical details
- **Don't assume**: Verify all findings with evidence
- **Don't skip steps**: Playbooks exist for a reason
- **Don't alert attacker**: Avoid actions that reveal investigation
- **Don't work in silo**: Communicate with team about findings
- **Don't forget context**: Consider business operations and approved activities

## Common Investigation Mistakes

1. **Confirmation Bias**: Looking only for evidence that confirms initial assessment
   - **Fix**: Actively look for evidence contradicting your theory

2. **Insufficient Correlation**: Investigating alert in isolation
   - **Fix**: Always check multiple log sources and related timeframes

3. **Premature Closure**: Closing alert before complete investigation
   - **Fix**: Follow playbook completely, even for "obvious" alerts

4. **Poor Documentation**: Incomplete or unclear investigation notes
   - **Fix**: Document as if someone else will review your work

5. **Missing IOC Extraction**: Failing to collect all indicators
   - **Fix**: Create checklist for each alert type
