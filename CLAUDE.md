# GitHub Actions Security
**CRITICAL**: Always pin third-party actions to **commit SHAs**, not version tags. Version tags (e.g., `@v4`) can be hijacked in supply chain attacks.
```yaml
# Good — immutable
uses: actions/checkout@34e114876b0b11c390a56381ad16ebd13914f8d5  # v4

# Bad — mutable tag, vulnerable to hijacking
uses: actions/checkout@v4
```
To find the SHA for a tag: `gh api repos/OWNER/REPO/git/ref/tags/TAG --jq '.object.sha'` (dereference annotated tags with a second call if `.object.type` is `"tag"`).

# Internal HTTP Calls (Sifnos Envelope)
All REST API responses use the Sifnos envelope: `{"data": <payload>, "meta": {...}}`. When calling our own internal APIs from backend code, **always use `InternalAsyncClient`** (from `analysi.common.internal_client`) instead of bare `httpx.AsyncClient`. It auto-unwraps the envelope so `.json()` returns the payload directly. Bare `httpx.AsyncClient` is only for external third-party API calls. A lint check (`scripts/code_quality_tools/detect_bare_httpx.py`) enforces this.

# Environment Variable Naming Convention

**Product Name**: Analysi (the security automation platform)

Use this naming pattern for all environment variables:
```
ANALYSI_{CONTEXT}_{COMPONENT}_{PURPOSE}
```

Where:
- `ANALYSI` - Product name prefix (always use this)
- `CONTEXT` - When/where the variable is used:
  - Runtime contexts: No additional prefix needed
  - Build/packaging contexts: Use `BUILD_` prefix
- `COMPONENT` - What part of the system (e.g., `ALERT_PROCESSING`, `INTEGRATIONS`, `WORKFLOW`)
- `PURPOSE` - What the variable configures (e.g., `AGENT_DIR`, `RUNBOOKS_REPO`, `TIMEOUT`)

**Examples:**
- `ANALYSI_ALERT_PROCESSING_AGENT_DIR` - Runtime agent directory for alert processing

**Bad examples (avoid):**
- `AGENT_DIR` - Too generic, doesn't indicate product or context
- `RUNBOOKS_REPO` - Missing product prefix, unclear if build or runtime

# Docker Python Version

**CRITICAL**: All Docker containers MUST use the same Python version. Currently: `python:3.13-slim`

All services use the unified Dockerfile at `deployments/docker/Dockerfile`. The echo-server has its own Dockerfile at `deployments/docker/echo-server/app/Dockerfile` and must also stay in sync.

**Why**: The Cy language interpreter (`cy-language` package) compiles scripts to Python bytecode. A version mismatch causes "invalid syntax" errors at runtime.

# Worktree Isolation (AI Agents)

**CRITICAL**: If you are running in a git worktree, you MUST use Kind/Kubernetes. Docker Compose is blocked in worktrees.

**Main checkout** uses Docker Compose for interactive dev and demos. **Worktrees** (AI agents) use Kind exclusively to avoid polluting the shared Postgres and causing port conflicts.

Compose targets (`make up`, `make down`, `make restart`, `make logs`, etc.) will refuse to run in a worktree and print an error directing you to Kind.

## Kind Commands (use these in worktrees)
```bash
make k8s-up        # Full setup: create cluster + build images + deploy
make k8s-down      # Delete the Kind cluster
make k8s-build     # Rebuild images and load into Kind
make k8s-deploy    # Helm upgrade only (fast, no rebuild)
make k8s-status    # Show pod status and access URLs
make k8s-logs      # Tail logs (add SERVICE=api for specific service)
make k8s-verify    # Health check all pods + API
```

Each worktree gets a unique Kind cluster (`analysi-<worktree-slug>`) with unique ports. Up to 3 worktree clusters can run simultaneously. Run `make k8s-status` to see your assigned ports.

# Branch Integration Test Database

**CRITICAL**: Before running integration tests in any branch or worktree, you MUST start an isolated test database:

```bash
make test-db-up           # Start per-branch Postgres container + run Flyway migrations (~25s)
make test-integration-db  # Run DB-only integration tests
```

This creates a lightweight Postgres container specific to your branch (same `analysi-postgres:15-partman` image as production). Connection info is written to `.env.test.local` (gitignored). Each branch gets a deterministic port, so multiple branches can run test DBs simultaneously.

Integration tests will **fail with a clear error** if `.env.test.local` is missing — they will NOT silently fall back to the main Docker Compose database.

## Two tiers of integration tests

| Command | What it runs | What it needs |
|---------|-------------|---------------|
| `make test-integration-db` | DB-only tests (~1800+) | `make test-db-up` only |
| `make test-integration-full` | ALL integration tests (~2000+) | Full stack (compose or Kind) |

Tests that need Vault, Valkey, or MinIO are marked `@pytest.mark.requires_full_stack` and excluded from `test-integration-db`. When writing new integration tests that use these services, add the marker:

```python
pytestmark = [pytest.mark.integration, pytest.mark.requires_full_stack]
```

```bash
# When done with the branch
make test-db-down    # Stop and remove the container + .env.test.local
```

**Do NOT** attempt to use the main Docker Compose Postgres (port 5434) for integration tests in branches. The safety guard in `conftest.py` blocks this.

Unit tests (`tests/unit/`) do not need a database and work without `make test-db-up`.

# Database Migrations
CRITICAL: Flyway migration files are IMMUTABLE once applied:
- **NEVER edit SQL migration files** after they have been applied to the database
- If you need to change migration logic, create a NEW migration file with a new version number
- Use `make flyway-repair` if migrations are accidentally deleted or modified
- Migration files in `migrations/flyway/sql/` are executed in order and checksummed

# Testing Strategy
CRITICAL: NEVER use SQLite for tests. Our testing strategy is:
- **ALL tests use PostgreSQL**: Both unit and integration tests use the `analysi_test` database
- **Unit tests**: Use PostgreSQL with proper isolation (clean data between tests)
- **Integration tests**: Use PostgreSQL with full schema validation
- **SQLite is not supported**: SQLite does not support PostgreSQL features like ARRAY types, UUIDs, ENUM types, etc. that our models require
- **Test database**: Automatically created via Docker init scripts in `deployments/docker/configs/postgres/init/`

- never expect NotImplementedError in unit tests. Let them fail as per TDD guideline
- never do `pytest.raises(NotImplementedError):`
- When writing documentation, prefer to be short and focus on the most important aspects. No need to be comprehensive.
- Always add `@pytest.mark.integration` for integration tests
- use "poetry run ruff check --fix" as needed
- Always read related code (existing code) before trying to invent a new design
- All our REST APIs should have at least one integration test for all common operations.
- do not name unit tests with their implementation phase. test_phase_6_1_web_app_apis.py is a bad name.
- Always use timezone aware timestamps in this project

# Partition Management (pg_partman)
Partitions are managed by **pg_partman** (configured in Flyway migration V094). No hand-rolled Python partition code — pg_partman handles creation, cleanup, and retention automatically.

## Architecture
- **pg_partman**: Creates future partitions and drops expired ones via `run_maintenance_proc()`
- **pg_cron**: Runs maintenance hourly (configured in V094 migration)
- **pg_partman BGW**: Background worker as safety net (checks every 30 min)
- **Default partitions**: Every table has a `_default` partition to catch out-of-range inserts

## Key Files
- `deployments/docker/postgres/Dockerfile` — Custom PG image with pg_partman + pg_cron
- `migrations/flyway/sql/V094__setup_pg_partman.sql` — Registers all tables, sets retention
- `src/analysi/services/partition_management.py` — Thin wrapper (run_maintenance, check_health)
- `tests/utils/db_cleanup.py` — Test cleanup (TRUNCATE) and partition validation
- `deployments/helm/analysi/templates/partition-retention-configmap.yaml` — Helm retention override

## Usage
```bash
# List partition counts per table
make list-partitions

# Validate partition counts within safety limits
make validate-partitions

# Check for data in default partitions or gaps
make partition-health

# Trigger pg_partman maintenance on demand
make partition-maintenance

# Emergency: drop ALL partitions (requires confirmation)
make emergency-cleanup
```

## Retention (configurable via Helm)
| Tables | Default | Override via |
|--------|---------|-------------|
| task_runs, artifacts, workflow_*, integration_runs, control_events | 90 days | `postgresql.partitioning.retention.task_runs` |
| alerts, alert_analysis | 180 days | `postgresql.partitioning.retention.alerts` |
| activity_audit_trail | 365 days | `postgresql.partitioning.retention.activity_audit_trail` |

## Important Notes
- **Never manually CREATE/DROP partitions** — pg_partman manages the lifecycle
- **Default partition data = bug** — if `make partition-health` shows data in default partitions, a partition is missing
- **Partition counts are validated** on every test run (safety limit: 100 per table)
- **New partitioned table?** Add it to V094 migration pattern: call `partman.create_parent()` in a new migration
- **pg_partman naming**: New partitions use `_pYYYY_MM_DD` format; old ones keep `_YYYY_MM_DD`
- Any time we create a new MCP tool or REST API, we should decide if it needs to be logged in the audi trail (when used that is)
- Whenever we modify the REST API or MCP API we need to restart the server to take a effect `make restart-api`.

# Human-in-the-Loop (HITL)

## Architecture
Cy scripts pause at hi-latency tools (e.g., `ask_question_channel`). Pause propagates: Task → Workflow → Analysis. Resume propagates back down via `human:responded` control events.

## Key Patterns
- **Three-layer pause**: `TaskRun.status=paused` → `WorkflowNodeInstance.status=paused` → `AlertAnalysis.status=paused_human_review`
- **Memoized Replay**: `ExecutionCheckpoint` stores `node_results`, `pending_tool_args`, `variables`. On resume, completed nodes are replayed from cache (no LLM re-calls).
- **`pending_tool_result` is None on initial pause**: The hi-latency tool does NOT execute before the interpreter pauses. The backend sends the Slack message itself via `send_hitl_question()`.
- **Question tracking**: `hitl_questions` table (partitioned monthly by `created_at`). Matches Slack button clicks via `(question_ref, channel)` = `(message_ts, channel_id)`.
- **Audit trail**: HITL answers are logged as `hitl.question_answered` activity events.

## File Locations
| Component | Path |
|-----------|------|
| HITL constants | `src/analysi/constants.py` → `HITLQuestionConstants` |
| Question model | `src/analysi/models/hitl_question.py` |
| Question repository | `src/analysi/repositories/hitl_repository.py` |
| Slack listener service | `src/analysi/slack_listener/service.py` |
